## 1. Architecture Map

### 1a. IQMainGPU.cs (Gold Standard)

**OnStateChange → OnBarUpdate → OnMarketDepth → OnRender pipeline**

| Stage | Responsibility | State mutated |
|-------|---------------|---------------|
| `SetDefaults` | Sets 16 property groups, registers 7 plots (EMA5/13/50/200/800 + 2 cloud bands) | All property backing fields |
| `Configure` | Upgrades `MaximumBarsLookBack` to `Infinite` for ≥60-min and Day/Week/Month bars | `MaximumBarsLookBack` |
| `DataLoaded` | Allocates all collections with bounded initial capacities; caches EMA/StdDev indicator refs; initialises alert flags | All private state |
| `Terminated` | Calls `DisposeDXResources()` | Frees all DX objects |
| `OnBarUpdate` | Computes sessions, pivots, ranges (ADR/AWR/AMR/RD/RW), VWAP, PVSRA candles, microstructure, OTE zones; pre-builds `dashLine2`–`dashLine7` strings | ~30 scalar fields + all collection contents |
| `OnMarketDepth` | Updates `bidBook`/`askBook`; calls `DetectOrderBookWalls`; writes `l2StatusText` | `bidBook`, `askBook`, `l2StatusText`, `level2Available` |
| `OnRenderTargetChanged` | Sets `dxReady = false`; calls `DisposeDXResources()` | DX resources released |
| `OnRender` | Calls `CreateDXResources()` when `!dxReady`; dispatches 12 Render* methods each in `try { } catch(SharpDXException) { dxReady=false; Dispose; return; } catch(Exception) { log; }` | Writes only to the render target |

**SharpDX / Direct2D resource lifecycle**

- **Created**: `CreateDXResources()`, called lazily from `OnRender` when `!dxReady`. Pattern: call `DisposeDXResources()` first → allocate all objects → set `dxReady = true` at end; catch-all sets `dxReady = false` and re-calls `DisposeDXResources()`.
- **Disposed**: `DisposeDXResources()`, called from `State.Terminated`, `OnRenderTargetChanged`, and on `SharpDXException` during render. Uses a private `DisposeRef<T>` helper that nulls each field after disposal. Every object allocated in `CreateDXResources` has a corresponding `DisposeRef` call — **no leaks identified**.
- `MakeBrush` helper wraps non-frozen WPF brushes with `try/catch(InvalidOperationException)` on `.Color` access — correct per NT8 SharpDX conventions.

**Collection growth/eviction policy**

| Collection | Cap | Eviction strategy |
|---|---|---|
| `snapshots` | 1,000 | `RemoveAt(0)` |
| `deltaHistory` | 1,000 | `RemoveAt(0)` |
| `srLevels` | 50 | `RemoveAt(0)` |
| `sessionBoxes` | 200 | `RemoveAt(0)` |
| `dailyOpenEntries`, `ethOpenEntries`, `rthXxxOpenEntries` | 200 each | `RemoveAt(0)` |
| `vwapEthData`, `vwapRthData` | 1,000 | `RemoveAt(0)` |
| `liquidityZones` | `MaxActiveZones` (default 100) | Evicts oldest recovered zone; if none, oldest zone |
| `oteZones` | `OTEMaxZones` (default 3) | `RemoveAt(0)` |
| `dailyRanges`/`rdRanges` | `AdrLength`/`RdLength` | `Dequeue()` |
| `weeklyRanges`/`rwRanges` | `AwrLength`/`RwLength` | `Dequeue()` |
| `monthlyRanges` | `AmrLength` | `Dequeue()` |
| **`bidBook`, `askBook`** | **Unbounded** | **Never pruned — see §5a / E11** |

**Threading**

NT8 dispatches all callbacks (including `OnMarketDepth`) on the UI thread when the indicator is live. `bidBook`/`askBook` are written in `OnMarketDepth` and read in `OnRender` — both on the same thread, no race. `sessionBoxes` is accessed under `lock(_sessionLock)` in `OnRender` (via `ToList()`) but written from `OnBarUpdate` without the lock. In practice NT8 serialises `OnBarUpdate` and `OnRender` on the UI thread, so the lock is redundant but harmless — the unsynchronised write is a latent issue if NT8 ever allows concurrent callbacks.

### 1b. IQMainGPU_Enhanced.cs

Architecture is identical to Main, with these additions:

**New pipeline stage inside `OnRender`** — after step 11 (DST table), step 12 dispatches to `RenderDashboard` which:
1. Computes `needMetrics` (true if Monitoring or Entry dashboards are not hidden).
2. Calls `CalculateDashboardMetrics()` exactly once if needed.
3. Dispatches to `RenderMainDashboard`, `RenderMonitoringDashboard`, `RenderEntryModeDashboard`, and `DrawStopTargetLinesOnChart`.

**Additional state**:
- Pre-built `dashLine2`–`dashLine7` string fields, set in `UpdateDashboardLines()` called from `OnBarUpdate` — dashboard strings are built outside the render thread.
- `dashboardPrimarySignal`, `dashboardStopPrice`, `dashboardTargetPrice`, `dashboardConfidence`, `dashboardConflictDetected`, `dashboardConflictText` — computed once per frame in `CalculateDashboardMetrics()` (line ~3675).
- `lastSignalDetectedTime`, `lastTrackedSignal`, `signalIsStale` — wall-clock signal staleness tracking.

**No new unbounded collections.** All inherited caps unchanged.

### 1c. IQMainUltimate.cs

Architecture is Enhanced + TPO engine layer:

**TPO state** (~lines 430–510):
- `tpoSessions` (`List<TPOSession>`, capped at 200, shared under `_sessionLock`)
- `activeTPOSessions[8]` — one live session per session ID
- `previousDayTPO` — last finalized London/NY session
- Scalar cache: `tpoCurrentPOC`, `tpoCurrentVAH`, `tpoCurrentVAL`, `tpoCurrentIBHigh/Low`, `tpoCurrentIBHighExt/LowExt`, `tpoCurrentShape`, `tpoCurrentSessionLabel`, `tpoNearestNakedLevel`
- `nakedTPOLevels` (`List<NakedTPOLevel>`, capacity `MaxNakedLevels + 5`)
- `_usedLabelYPositions` (`HashSet<int>`) — label collision avoidance for TPO renders

**OnBarUpdate** calls `UpdateTPOSession(barUtc)` inside `IsFirstTickOfBar` guard at line 2818. All TPO sub-methods are invoked transitively. ✓

**OnRender** adds `RenderTPOLevels` call at line 3245, guarded by `ShowPOC || ShowValueArea || ShowInitialBalance || ShowNakedLevels`. `_usedLabelYPositions.Clear()` is called at line 3146, unconditionally at the **top** of `OnRender` before any render methods — collision state does not leak across frames. ✓

**Unique improvement over Enhanced**: `_latestClose`, `_cachedVwapValue`, `_cachedEma50Value` are cached in `OnBarUpdate` (line 2540) and used by `CalculateDynamicStop/Target/EntryScore` — preventing stale-data hazards where an `OnRender` call might read `Close[0]` while bar state is advancing. Enhanced reads `Close[0]` directly in render-time computations.

**`MaxTPOStopTicks` / `MaxTPOTargetTicks`** are `private const int = 200/400` (lines 440–441) — not user-configurable properties. See §4 / §8 for the product question.

## 3. NT8 Lifecycle Correctness

### IQMainGPU.cs — Reference-quality

- All `[NinjaScriptProperty]` defaults set in `State.SetDefaults`. ✓
- `IsOverlay = true`, `DrawOnPricePanel = true`, `Calculate = OnEachTick`, `MaximumBarsLookBack` upgraded in `Configure` for long periods. ✓
- `OnBarUpdate` lookback guards: `CurrentBar >= PVSRALookback`, `CurrentBar >= 50` for EMA50, `CurrentBar >= DeltaLookback` for delta. ✓
- L2: `OnMarketDepth` fast-returns if `!EnableLevel2`. No `AddDataSeries` for L2 needed — NT8 `OnMarketDepth` override is sufficient. ✓

### IQMainGPU_Enhanced.cs — Two issues

1. **Default `Calculate = Calculate.OnEachTick`** (line 1776) contradicts the documentation (`IQMainGPU_Enhanced.md` recommends "On price change"). Users accepting defaults will run at highest CPU load.
2. **No `[Range]` on dashboard font size properties** (`MainDashboardFontSize`, `MonitoringDashboardFontSize`, `EntryModeDashboardFontSize`) — a user typing 0 or a negative number would cause `CreateDXResources` to throw (DirectWrite rejects ≤0 font size). Similarly `DashboardOpacity` has no `[Range(1, 100)]`.

### IQMainUltimate.cs — Same as Enhanced + minor

- `SignalStaleMinutes`, `SignalExpireMinutes` have no explicit `[Range]` attribute despite being user-editable integers (defaults 15/30). Setting to 0 would flag every signal as stale/expired immediately.
- `MaxTPOStopTicks` and `MaxTPOTargetTicks` are hardcoded `private const` — not user-configurable (see §8 for product question).
- When `previousDayTPO == null` (neither London nor NY sessions enabled), all TPO-based stop/target modes silently fall back to the ADR-based calculation without any warning logged. This is silent degradation.

### Group numbering inconsistency

| Group | Main | Enhanced | Ultimate |
|-------|------|----------|---------|
| 12 | `"12. Dashboard"` (ShowDashboard, DashPos, DashFontSize, DashOpacity, ShowHalo, colors) | `"12. Halo & Candle Colors"` (halo/colors only; old dashboard props removed) | `"12. Signal Colors & Halo"` (halo/colors; DashFontSize/DashOpacity moved to group 13) |

| 16 | Not present | `"16. Dashboards"` ✓ | `"16. Dashboards"` ✓ |
| 17 | Not present | Not present | `"17. TPO Settings"` |

**Problem**: The `IQMainGPU_Enhanced.md` states "groups 1–15 are identical to IQMainGPU." This is false for group 12. Main's group 12 contains `ShowDashboard`, `DashPosition`, `DashFontSize`, `DashOpacity`; Enhanced's group 12 contains only the halo/color properties. A user migrating from Main finds the dashboard controls in a different group under a different name.

### Legacy / dead properties in Ultimate

**Lines 1535–1544** in Ultimate:
```csharp
[NinjaScriptProperty]
[Browsable(false)]
public bool ShowDashboard { get; set; }          // set in SetDefaults, never read in rendering

[NinjaScriptProperty]
[Browsable(false)]
public IQMDashboardPosition DashPosition { get; set; }   // same
```
These are correctly initialised in `SetDefaults` (lines 2297–2298) for XML serialization compat with IQMainGPU templates. They are **never read** in any rendering or computation path in Ultimate. This is by design but the design intent is only in a comment — not surfaced to the user.

**Removing them** would invalidate serialized templates from IQMainGPU that contain `<ShowDashboard>` or `<DashPosition>` XML elements. NT8 would silently ignore the unrecognized elements, generating no crash but potentially a deserialization warning. **Verdict: keep them.**

**Enhanced has NO equivalent backward-compat stubs** — a user who had a saved IQMainGPU template and applies it to Enhanced will silently lose the old `ShowDashboard`/`DashPosition` bindings. This is an Enhanced-only migration gap that Ultimate correctly handles.

A user sees two different dropdown types for visually similar "position this overlay" controls. There is no way to hide the Range Table (only the DST Table has a `ShowDstTable` toggle). The inconsistency exists across both Enhanced and Ultimate.

## 5. Per-Tier Flaw List

### 5a. Enhanced — What Makes It "Not Actually Enhanced"

**E1. No dashboard auto-stacking — collision guaranteed with same-corner placement**
`GetDashboardPosition` (lines ~4035–4070) assigns each dashboard an independent pixel origin from its `DashboardPositionType`. If two or more dashboards share the same position (e.g., Main=`TopLeft` and Monitoring=`TopLeft`), they paint on top of each other with no offset or stacking. The documentation warns about price-axis overlap on the right, but says nothing about same-corner mutual overlap.

**E2. `CalculateDashboardMetrics` called once per frame — CORRECT ✅**
`RenderDashboard` (~line 3652) checks `needMetrics` exactly once and calls `CalculateDashboardMetrics()` a single time before dispatching to all three Render* methods.

**E3. `CalculateDynamicStop` is direction-blind — stop placed on wrong side for bearish setups (lines 3768–3807)**
All four `StopMode` branches (`AutoDetected`, `PivotBased`, `HVNBased`, `ManualInput`) unconditionally place the stop **below** price (`close - ...`), assuming a bullish entry. When `GetPrimarySignal()` returns "BEARISH ABSORPTION", "BEARISH IMBALANCE", "SELL PRESSURE", or "FAKE BREAKOUT", the stop is on the wrong side of price — a short-seller would be stopped into a loss by a normal uptick.

Additionally, `AutoDetected` iterates `liquidityZones` and returns the **first** bullish zone below price in insertion order — not the **nearest** one. A zone from 100 bars ago could be returned before a fresh nearby zone.

**E4. `CalculateDynamicTarget` is direction-blind (lines 3810–3837)**
All modes return resistance levels **above** price (`R1`, `R2`, nearest SR above). For bearish signals the target should be below price (`S1`, `S2`). A bearish trader using `AutoDetected` target mode is handed a resistance level — the opposite of a profit target.

**E5. R/R calculation is bullish-only (lines 4115–4120)**
```csharp
if (dashboardStopPrice > 0 && dashboardEntryPrice > dashboardStopPrice)
{ ... calculate rr ... }
```
The condition `entryPrice > stopPrice` is only true for long setups. For bearish setups where stop > entry, `rr` stays 0 and the dashboard always shows "R/R: N/A" — a direct consequence of E3/E4.

**E6. `DrawStopTargetLinesOnChart` draws lines regardless of signal state (lines 4289–4311)**
Lines (entry/stop/target) are drawn whenever `ShowStopTargetLines && ShowEntryModeDashboard && position != Hidden`, with no check for `signalExpired` or `noSignal`. Once lines appear, they persist on-chart indefinitely — including after the signal expires and the dashboard shows "No Active Signal". The entry line also permanently tracks `Close[0]`, redrawing on every tick as a horizontal chasing line.

**E7. `DetectConflicts` triggers a near-permanent "conflict" for off-hours (lines 4009–4026)**
The "Low Participation Session" check fires `anyConflict = true` any time outside London, NY, EU Brinks, and US Brinks. These four sessions cover roughly 13–14 hours per day; the remaining 10–11 hours are perpetually flagged as "LOW PARTICIPATION SESSION" conflict. On weekday nights this fills the Entry Mode and Monitoring dashboards with a persistent yellow warning. The documentation acknowledges this and suggests disabling conflict warnings or reducing verbosity, but there is no per-conflict-type toggle.

**E8. `STALE_SIGNAL_MINUTES` and `EXPIRE_SIGNAL_MINUTES` are hardcoded private constants (lines 321–322)**
```csharp
private const int STALE_SIGNAL_MINUTES  = 15;
private const int EXPIRE_SIGNAL_MINUTES = 30;
```
These cannot be changed by the user. Ultimate correctly exposes them as `[NinjaScriptProperty]` `SignalStaleMinutes` / `SignalExpireMinutes` (lines 1967–1975, defaults 15/30). Enhanced should match.

**E9. `WrapTextToLines` text overflow when panel is narrowed by `ClampTableY` (lines 4135–4143, 4389–4421)**
In `RenderEntryModeDashboard` (and `RenderMonitoringDashboard`), the conflict text is wrapped with:
```csharp
float usableW = panelW - PadX * 2;           // panelW is the original 420f, pre-clamp
foreach (string wrapped in WrapTextToLines(trimmed, usableW, EntryModeDashboardFontSize))
```
But `panelW` may be reduced by `ClampTableY` two lines later. `WrapTextToLines` has already been called with the wider width, so `charsPerLine` is computed for 420px even if the panel was clamped to 300px. Text overflows the right edge of the clamped panel.

**E10. No `ShowDashboard`/`DashPosition` backward-compat stubs (no lines)**
Enhanced does not declare `[NinjaScriptProperty] [Browsable(false)]` stubs for the old IQMainGPU group-12 `ShowDashboard` / `DashPosition` properties. Users loading an IQMainGPU saved template onto Enhanced get a silent XML deserialization mismatch. Ultimate handles this correctly with stubs at lines 1535–1544.

**E11. `bidBook`/`askBook` grow unbounded (lines 2189–2190 DataLoaded)**
These dictionaries are initialized with capacity 200 but are never pruned. On heavily-quoted instruments (bonds, currency futures) with L2 enabled, every price level ever seen adds an entry. Over a trading session this can accumulate tens of thousands of entries, increasing `RenderWallLines` iteration time and GC pressure linearly. Main shares this same flaw.

**E12. LINQ `Count(predicate)` in `RenderMainDashboard` hot path (line 3353)**
```csharp
int nonEmpty = lines.Count(s => !string.IsNullOrEmpty(s));
```
`Count(predicate)` on an array uses `IEnumerable<T>.Count` extension — allocates an enumerator object on every render frame. Should be a plain `for` loop counter.

### 5b. Ultimate — What Makes It "Not Actually Ultimate"

**U1. TPO engine IS fully wired — confirmed ✅**
`UpdateTPOSession(barUtc)` called at line 2818; `RenderTPOLevels` called at line 3245. All sub-methods (`CalculatePOC`, `CalculateValueArea`, `UpdateInitialBalance`, `CalculateIBExtensions`, `ClassifyProfileShape`, `CreateNakedTPOLevels`, `UpdateNakedLevels`, `CleanupOldNakedLevels`, `UpdateNearestNakedLevel`) are reachable and exercised. This is the headline Ultimate feature and it works.

**U2. `CalculateDynamicStop`/`Target` honour all TPO modes — confirmed ✅**
`UltimateStopMode.TPOBased`, `AutoDetected`, `PivotBased`, `HVNBased`, `ManualInput` all correctly check `IsBearishSignal()` and place stops/targets on the appropriate side of price. `MaxTPOStopTicks`/`MaxTPOTargetTicks` caps are enforced with `Print()` diagnostics when skipped. Direction-awareness is fully correct — this fixes Enhanced's E3/E4/E5.

**U3. `IsBullishSignal`/`IsBearishSignal` are consistent with `GetPrimarySignal` ✅**
- `GetPrimarySignal` returns: "FAKE BREAKOUT", "BULLISH ABSORPTION", "BEARISH ABSORPTION", "BULLISH IMBALANCE", "BEARISH IMBALANCE", "BUY PRESSURE", "SELL PRESSURE", "NEUTRAL", "No Data"
- `IsBullishSignal`: `Contains("BULLISH") || == "BUY PRESSURE"` — covers all three bullish values ✓
- `IsBearishSignal`: `Contains("BEARISH") || == "SELL PRESSURE" || == "FAKE BREAKOUT"` — covers all four bearish values ✓
- "NEUTRAL"/"No Data" → neither → fallback to manual tick distance ✓

**U4. `GetNonCollidingLabelY` is reset per frame ✅**
`_usedLabelYPositions.Clear()` at line 3146, executed unconditionally at the top of `OnRender` before any render method is called. No state leaks across render frames.

**U5. `MaxTPOStopTicks`/`MaxTPOTargetTicks` are not user-configurable (lines 440–441)**
```csharp
private const int MaxTPOStopTicks   = 200;
private const int MaxTPOTargetTicks = 400;
```
For small-tick instruments (CL at $0.01/tick, 200 ticks = $2.00) the cap of 200 ticks is extremely tight and will reject almost every valid TPO stop, silently falling back to the ADR-based distance with no user notice except a `Print()` log. For NQ at 0.25/tick, 200 ticks = 50 points which is generous. This is an asset-class-specific calibration issue. **Should be a user-configurable `[NinjaScriptProperty]` in group 16 or 17.**

**U6. `previousDayTPO` is `null` when neither London nor NY session is enabled**
`FinalizeTPOSession` sets `previousDayTPO` only for session IDs 0 (London) or 1 (NY) (line 5771). If a user disables both sessions, `previousDayTPO` stays null indefinitely and all "previous day VAH/VAL" lookups in `CalculateDynamicStop/Target` silently skip without logging. The `Print()` diagnostics for level-skipping only fire when a level exists but exceeds the tick cap — they do not fire when `previousDayTPO == null`.

**U7. Dashboard auto-stacking not inherited — same collision risk as Enhanced**
Ultimate's `RenderDashboard` method (lines ~3985–3997) uses the exact same `GetDashboardPosition` + independent-origin logic as Enhanced. No auto-stacking was added. Three dashboards in the same corner still overlap silently.

**U8. `CalculateIBExtensions` called twice per session — redundant but harmless**
Called once in `UpdateInitialBalance` at the first bar after the 60-minute IB window closes (line ~5868), and again in `FinalizeTPOSession` (line ~5766). The second call recalculates identical values. No side effect, but wastes a few CPU cycles.

**U9. `tpoSessions` iteration under `lock(_sessionLock)` in `OnRender` (lines 6079–6139)**
Both `UpdateTPOSession` (OnBarUpdate) and `RenderTPOLevels` (OnRender) acquire `_sessionLock`. Since both run on the UI thread in NT8, the lock cannot cause a deadlock, but it does mean `RenderTPOLevels` and `UpdateTPOSession` cannot interleave — which is the correct intent, but the implementation comment and the Main file's usage of the same lock pattern suggest this was cargo-culted from the session boxes locking pattern where it made more sense. Net effect: harmless.

**U10. TPO bin resolution uses raw `TickSize` — not asset-class-adjusted**
The `UpdateTPOSession` bucket key is `Math.Round(typicalPrice / tickSz) * tickSz` with `tickSz = TickSize` (defaulting to 0.25 if ≤0). For instruments with very fine ticks (CL: 0.01, bonds: 1/64), a 500-tick session produces 5,000 dictionary buckets. Across 200 historical sessions × 8 session IDs in `tpoSessions`, worst case is ~8 million dictionary entries. In practice `tpoSessions` is capped at 200 and most futures instruments have reasonable tick sizes, so OOM is unlikely. However for CL users this should be benchmarked.

**U11. `nakedTPOLevels` list grows up to `MaxNakedLevels + 3` before pruning**
`CreateNakedTPOLevels` adds up to 3 entries (POC + VAH + VAL) then prunes with `while (count > MaxNakedLevels) RemoveAt(0)`. The list briefly holds up to `MaxNakedLevels + 3` entries between the add and the while-loop. With default `MaxNakedLevels = 10`, the list temporarily holds 13. This is cosmetically minor — no real growth issue.

**U12. `DrawStopTargetLinesOnChart` inherited unchanged — still draws when signal expired**
Ultimate inherits Enhanced's `DrawStopTargetLinesOnChart` verbatim. The bug (E6) — lines persist after signal expires — is not fixed in Ultimate.

**U13. `WrapTextToLines` pre-clamp width bug inherited from Enhanced**
Ultimate's `RenderEntryModeDashboard` and `RenderMonitoringDashboard` contain the same pre-clamp `WrapTextToLines` call (lines ~4861, ~5015). Enhanced's E9 is not fixed in Ultimate.

## 8. Open Questions for the User

1. **Dashboard auto-stacking**: When two dashboards share a corner, should the second dashboard stack immediately below the first (auto-stacking), or should the user be responsible for choosing distinct corners? Or should a collision warning appear in the status bar?

2. **Low Participation conflict**: Should "Low Participation Session" be its own boolean toggle (default: off during Tokyo/Sydney/Frankfurt, on during pre-market)? Or should the user simply set `ConflictLevel = Brief` or `ShowConflictWarnings = false`?

3. **`MaxTPOStopTicks` / `MaxTPOTargetTicks`**: Should these be user-configurable properties (recommended: yes), and what defaults make sense per asset class — or should they stay as one global value with a recommended note per instrument?

4. **Default `Calculate` mode for Enhanced**: Should the default be `Calculate.OnEachTick` (maximum responsiveness, highest CPU) or `Calculate.OnPriceChange` (matches documentation, lower CPU)? This affects how the indicator ships for new users.

5. **Range Table / DST Table positioning**: Should `TablePosition` and `DstTablePosition` be migrated to `DashboardPositionType` (gaining `Hidden` and `CenterTop`/`CenterBottom` options), or should the old `IQMDashboardPosition` enum be extended with `Hidden` only?

6. **`ShowRangeTable` with no `Hidden` option**: The Range Table has a `ShowRangeTable` bool but no positional `Hidden` option. Should a `Hidden` position in the enum replace the bool, or keep both?

7. **TPO Value Area percentage**: The `CalculateValueArea` method uses 70% of total volume as the Value Area definition. Should this be a user-configurable `ValueAreaPercent` property (typical market profile default: 70%), or stay hardcoded?

8. **TPO bin granularity for small-tick instruments**: For CL (tick = $0.01), should the bin size be auto-scaled to e.g. 4× tick size to reduce profile dictionary size, or should a `TPOBinSizeMultiplier` property be exposed?

9. **`previousDayTPO` session selection**: Currently only London (0) and NY (1) are used as reference sessions. Should this be configurable (e.g., a `TPOReferenceSessionId` property), or is London/NY hard-coding the correct institutional convention?

10. **`bidBook`/`askBook` max depth**: What is an acceptable cap for the order book dictionaries (e.g. 2000 price levels)? Eviction policy when full — drop farthest-from-price levels, or oldest-inserted?

11. **Stop/target lines when signal is NEUTRAL**: Should the lines disappear (current documented intent, not implemented), or should they hold at the last valid signal's stop/target prices until the next signal fires?

12. **Template migration from IQMainGPU**: For users migrating a saved workspace from IQMainGPU to Enhanced, the old `ShowDashboard = false` setting is lost and the new dashboards default to visible. Should there be a first-run migration message? Or is silent defaults-override acceptable?
